# GitHub Actions CI/CD 실습 가이드

## 개요

이 실습은 GitHub Actions를 통해 **CI(지속적 통합)**와 **CD(지속적 배포)**의 핵심 가치를 직접 체험하는 것을 목표로 합니다.

---

## 사전 준비

1. GitHub 계정이 있어야 합니다.
2. 실습용 저장소를 본인 계정으로 **Fork**합니다.
   - 조직 저장소 페이지 우측 상단 **Fork** 버튼 클릭
   - 본인 GitHub 계정을 owner로 선택 후 생성
3. Fork된 저장소를 로컬에 **Clone**합니다.

```bash
git clone https://github.com/YOUR_NAME/26SS-Session-github-action-demo.git
cd 26SS-Session-github-action-demo
npm install
```

4. Fork된 저장소의 **Actions 탭**을 활성화합니다.
   - GitHub 저장소 → **Actions 탭** 클릭
   - "I understand my workflows, go ahead and enable them" 버튼 클릭
5. **CD 배포 환경**을 생성합니다. (시나리오 3 사전 설정)
   - GitHub 저장소 → **Settings** → **Environments** → **New environment**
   - `staging` 생성 (보호 규칙 없음)
   - `production` 생성 → **Required reviewers**에 본인 계정 추가

## 실습 시나리오

### 시나리오 1: Branch Protection으로 코드 품질 강제

**목표**: CI를 통과하지 못한 코드는 main에 머지할 수 없도록 설정합니다. 이후 시나리오에서 Merge 차단을 직접 체험하려면 이 설정이 먼저 필요합니다.

**설정 방법**:

1. GitHub 저장소 → **Settings** → **Branches**
2. **Add branch protection rule** 클릭
3. Branch name pattern: `main`
4. 다음 항목 체크:
   - ✅ Require status checks to pass before merging
   - ✅ Require branches to be up to date before merging
   - Status checks 검색창에서 `테스트 (Node 20)`, `코드 스타일 검사 (ESLint)` 선택
5. **Save changes**

> Status checks 목록에 항목이 안 보인다면, 먼저 시나리오 2를 진행해 CI를 한 번 실행한 뒤 돌아와서 설정하세요.

**결과**: 이제 CI가 실패한 PR은 물리적으로 main에 머지 불가!

---

### 시나리오 2: CI가 자동으로 코드를 검사한다

**목표**: main 브랜치에 push하면 CI가 자동 실행되는 것을 확인합니다.

> Branch Protection의 Status checks에 항목이 없었다면, 이 시나리오 진행 후 시나리오 1로 돌아가서 설정을 완료하세요.

**단계**:
1. `src/calculator.js`에 아주 작은 변경을 가합니다. (예: 주석 추가)
2. push합니다.

```bash
git add src/calculator.js
git commit -m "docs: 주석 추가"
git push
```

3. GitHub 저장소 → **Actions 탭**으로 이동합니다.
4. `CI - 테스트 & 린트` 워크플로우가 자동으로 실행되는 것을 확인합니다.
5. 각 Job (린트 → 테스트 Node 18/20/22)이 순서대로 실행되는 것을 관찰합니다.

**확인 포인트**:
- 내가 아무것도 하지 않아도 테스트가 자동 실행됨
- 세 가지 Node.js 버전이 **동시에(병렬로)** 테스트됨
- 모든 체크가 통과하면 ✅ 초록 표시가 뜸

---

### 시나리오 3: CI가 버그를 자동으로 잡아낸다

**목표**: 코드에 버그를 넣었을 때 CI가 머지 전에 차단하는 것을 확인합니다.

**단계**:

1. 새 feature 브랜치를 만듭니다.

```bash
git checkout -b feature/buggy-subtract
```

2. `src/calculator.js`의 내용을 `src/calculator.buggy.js`의 내용으로 교체합니다.
   - `subtract` 함수에서 `-`가 `+`로 바뀐 버그가 있습니다.

3. push하고 PR을 만듭니다.

```bash
git add src/calculator.js
git commit -m "fix: subtract 함수 수정 (사실은 버그 도입)"
git push origin feature/buggy-subtract
```

4. GitHub에서 fork된 리포지토리의 main으로 향하는 **Pull Request**를 생성합니다. (원본 리포지토리로 올리지 않도록 주의!)
5. CI가 자동으로 실행되어 테스트 실패를 감지하는 것을 확인합니다.

**확인 포인트**:
- PR 페이지 하단에 ❌ 빨간 체크 표시가 나타남
- CI가 실패한 상태에서는 **Merge 버튼이 잠김** (Branch Protection 설정 시)
- 어떤 테스트가 왜 실패했는지 Actions 탭에서 확인 가능

**복구 방법**:
```bash
# 버그를 수정하고 다시 push
git add src/calculator.js
git commit -m "fix: subtract 함수 버그 수정"
git push
# → CI가 자동으로 다시 실행되어 통과!
```

---

### 시나리오 4: CD가 자동으로 배포한다

**목표**: main에 머지되면 자동 배포 파이프라인이 트리거되는 것을 확인합니다.

**단계**:

1. 시나리오 3에서 버그를 수정한 PR을 main에 **Merge**합니다.
2. **Actions 탭**에서 `CD - 자동 배포` 워크플로우가 자동 트리거되는 것을 확인합니다.
3. 파이프라인 흐름을 관찰합니다:

```
빌드 & 패키징
    ↓
스테이징 배포 (자동)
    ↓
[승인 대기] ← 실제 서비스에서는 여기서 사람이 승인
    ↓
프로덕션 배포
```

**확인 포인트**:
- 머지 한 번으로 전체 배포 파이프라인이 자동 실행됨
- 스테이징 → 프로덕션 순서가 보장됨
- 각 단계에서 무슨 일이 일어나는지 로그로 확인 가능

## 워크플로우 파일 구조 설명

```
.github/
└── workflows/
    ├── ci.yml        # CI: push/PR마다 자동 테스트 & 린트
    ├── cd.yml        # CD: main 머지 시 자동 배포
    └── pr-check.yml  # PR: PR 형식 자동 검사
```

### ci.yml 핵심 개념

| 개념 | 설명 |
|------|------|
| `on: push` | push 이벤트에 반응 |
| `on: pull_request` | PR 이벤트에 반응 |
| `jobs` | 병렬 또는 순차 실행 단위 |
| `needs` | job 간 의존성 (순서 보장) |
| `matrix` | 여러 환경을 동시에 테스트 |
| `uses: actions/checkout@v4` | 오픈소스 Action 재사용 |
| `cache: 'npm'` | 의존성 캐싱으로 속도 향상 |

### cd.yml 핵심 개념

| 개념 | 설명 |
|------|------|
| `environment` | 배포 환경 정의 (staging/production) |
| `concurrency` | 동시 배포 방지 |
| `workflow_dispatch` | 수동 실행 지원 |
| `upload-artifact` | job 간 파일 전달 |

## CI/CD 없이 vs. 있을 때 비교

| 상황 | CI/CD 없음 | CI/CD 있음 |
|------|-----------|-----------|
| 코드 리뷰 | 사람이 직접 실행해서 확인 | 자동으로 테스트 결과 표시 |
| 버그 발견 시점 | 배포 후 (이미 늦음) | PR 단계 (머지 전 차단) |
| 배포 방법 | 개발자가 수동으로 서버 접속 | 머지 버튼 클릭 한 번 |
| 배포 실수 | 잦음 (사람이 하므로) | 거의 없음 (자동화) |
| 새벽 긴급 배포 | 담당자 깨워야 함 | 자동으로 처리됨 |

## 자주 쓰는 GitHub Actions 마켓플레이스 Actions

```yaml
# 코드 체크아웃
uses: actions/checkout@v4

# Node.js 설치
uses: actions/setup-node@v4

# Python 설치
uses: actions/setup-python@v5

# 파일 캐싱
uses: actions/cache@v4

# 아티팩트 저장/불러오기
uses: actions/upload-artifact@v4
uses: actions/download-artifact@v4

# GitHub Pages 배포
uses: actions/deploy-pages@v4

# Slack 알림
uses: slackapi/slack-github-action@v1
```
